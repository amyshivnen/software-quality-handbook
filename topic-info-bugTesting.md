## Topic Info

### Key Guidelines (Do’s)

- Always log bugs clearly with steps to reproduce, expected result, and actual result
- Prioritise bugs based on how serious they are and how quickly they need to be fixed
- Use one system (e.g. GitHub Issues or Jira) so everything is tracked in one place
- Assign each bug to a specific person so it doesn’t get forgotten
- Fix bugs as early as possible to avoid bigger issues later
- Link bugs to commits or pull requests so changes can be tracked
- Include testing as part of development, not just something done at the end

### What to Avoid (Common Mistakes)

- Writing vague bug reports like “it doesn’t work”
- Leaving bugs unassigned or ignoring them
- Letting small bugs build up over time
- Mixing bugs with feature requests in the same list
- Only relying on manual testing
- Fixing bugs without testing them properly afterwards

### Workflow / Process

1. A bug is found (by a developer, tester, or user)
2. It is logged with clear details
3. The bug is prioritised and assigned
4. A developer fixes the issue
5. The fix is tested (manually or automatically)
6. The bug is marked as resolved

### Key Takeaways

- Clear bug tracking makes it easier for the team to stay organised
- QA is not just about fixing bugs, but preventing them
- Finding bugs early saves time and effort
- Having a consistent process reduces mistakes in production
